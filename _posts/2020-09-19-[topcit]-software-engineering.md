# 소프트웨어의 특성
- 비가시성
  - 다른 제품과 달리 눈에 보이지 않는다. 프로그램은 형태로 존재하지 않고 하드웨어에서 작동하기 때문이다.

- 복잡성
  - 소프트웨어는 매우 복잡할 뿐 아니라 만드는 방법과 과정 또한 복잡하다. 컴퓨터가 만들어진 초창기에 비해 소프트웨어는 나날이 복잡해지고 있다.

- 복제 가능
  - 컴퓨터 저장 장치의 공간만 있다면 소프트웨어를 복사 할 수 있다.~~국민학교 컴퓨터실 피카츄배구~~

- 테스트 및 변경 가능
  - 개발 과정 뿐만 아니라 출시 후에도 테스트가 진행된다. 그리고 문제가 발생하면 언제든지 수정이 가능하다.

- 비제조성
  - 별도의 생산 시설이 필요없다. 컴퓨터만 있으면 개발이 가능하다. 물론 임베디드 소프트웨어의 경우는 얘기가 다르다. 소프트웨어를 개발하기위해 무엇보다 필요한 것은 __두뇌력__ 이다.

- 순응성
  - 소프트웨어는 한 번 출시로 끝나지 않는다. 소프트웨어는 사용자들의 요구에 따라 계속해서 변화한다. 또는 하드웨어가 바뀜에 따라 최적화 될 수 있도록 변화해야한다.

# 소프트웨어 공학??
_ 아주 간단한 소프트웨어는 당신 혼자 만들 수도 있다. 꽤나 쓸만한 제품을 만들었더니 이용자가 많아진다. 이용자가 많아진만큼 이용자들이 요구하는 기능이 많아진다. 당신이 아무리 천재 개발자라도 혼자서는 기능과 규모를 관리하는데 한계가 생길 것이다. 이제 당신은 팀을 구성해서 협업을 하기 시작한다. 팀원들마다 각 각 잘하는 분야가 다르고 일정 관리도 해야한다. 할 일이 점점 많아진다. 연이어 후발주자들이 치고 올라온다. 속도 경쟁이 붙는다. 할 일은 많은데 시간은 부족하다. 집중력이 흐트러진다. 코드 한 줄 잘 못 써서 크리티컬한 버그가 생긴다. 누가 관리하던 부분인지 알 수도 없다.  이용자들이 떨어져나간다. 팀원들이 내 탓을 하는 것만 같다. 눈물이 앞을 가린다. _바이바이..._ 정신을 차려보니 상상이었다.

_ 소프트웨어를 성공적으로 개발하기까지 발생가능한 모든 어려움을 체계적이고 효율적으로 관리하기 위한 기술이 바로 __소프트웨어__ 공학이다. 당신이 좋은 개발자가 되고싶다면 소프트웨어 공학을 배워두자.

# 소프트웨어 공학의 변천사
_ 소프트웨어 공학은 하루 아침에 띡하고 나온게 아니다. 우리는 가끔 세상에 존재하는 것에 대해 너무 당연하게 생각하는 경향이 있는데 그런 습관은 버리는게 좋다. 적어도 무언가를 공부 할 때 만큼은 "아 이걸 내가 왜 배워야되냐고..."라고 생각하지말자. ~~내가 지금 그렇게 생각하고있다는 뜻은 아니다.~~

- [1950] 하드웨어 공학과 유사한 소프트웨어 공학 개념이 도입되기 시작함
- [1960] 소프트웨어에 대한 수요 급증, 인력 부족, __소프트웨어의 위기__ 발생, 소프트웨어 공학 본격 도입
- [1970] 소프트웨어에 대한 수요 급증, 전공 인력 부족, 비전공 인력 대거 투입, 소프트웨어의 수정이 쉽다는 점을 고안하여 만든 선코딩-후수정 방식 도입, 다양한 부작용 발생, 구조적 또는 정형적 기법들 발생, 분석-설계-구현 등을 순차적으로 진행하는 __폭포수 모델__ 개발
- [1980] 폭포수 모델은 비용이 많이 소요되고 진척도가 낮음, __재사용성__ 을 높여 개발 생산성을 높임
- [1990] 시장에서 경쟁우위를 선점하기위해 제품의 시장 출시 시간을 단축해야했음, 소프트웨어 개발 생산성의 대한 연구가 활발하던 시기, 폭포수 모델에서 요구사항, 설계 및 구현 등을 동시에 진행할 수 있는 __동시공학__ 에 집중한 모델을 사용
- [2000] 시장이 급속하게 변화하기 시작함, 효과적으로 대응하기위한 __에자일 방법론__ 도입

# 소프트웨어 개발 생명주기

1. 정의
  - 사용자 환경 및 문제점 이해에서 시작하여 운용/유지 보수에 이르기까지의 모든 과정
  - 타당성 검토 -> 개발계획 -> 요구사항 분석 -> 설계 -> 구현 -> 테스트 -> 운용 -> 유지보수\\

2. 목적
  - 프로젝트 비용 산정과 개발 계획 수립, 기본 골격 구성
  - 용어의 표준화
  - 프로젝트 관리

3. 소프트웨어 생명주기 선정
  - 기업에서 프로젝트의 개발 프로세스를 테일러링하는데 중요한 활동
  - 시스템 개발의 리스크와 불확실성 및 이에 대한 이해를 바탕으로 수행
  - 선택한 모델은 프로젝트에 존재하는 리스크/불확실성을 최소화시킬 수 있어야 함
  - 폭포수 모델, 프로토타입 모델, 진화 모델, 점증적 모델 등이 대표적인 생명주기 모델임\\

## 소프트웨어 생명주기 모델 종류
현업에서 가장 많이 사용되는 생명주기 모델을 살펴본다. 프로젝트 특성에 따라 생명주기 모델을 변경하여 사용 할 수도 있다.

1. V모델
![v-model](/assets/img/2020-09-19-[topcit]-software-engineering/v-model.jpg)
  - 관리자, 개발자 모두에게 수행되어야 할 활동 내역을 명확하게 보여준다.
  - 개발에 대해 잘 모르는 고객을 이해시키는 것이 용이하다.
  - 프로젝트에 적용, 관리하기가 용이하다.
  - 개발 활동의 시작/종류 조건 및 프로젝트를 효과적으로 관리 할 수 있는 척도 등이 명확하게 정의 될 수 있다.
  - 프로젝트의 검증 및 확인(Verification and Validation)을 강조하는 모델이다.
  - 요구사항 분석, 설계와 같은 개발 활동과 테스트 활동이 어떻게 연관 관계가 있는지를 잘 설명해준다.
  - 전체 개발 주기동안 개발 활동과 이에 해당하는 테스트 활동이 병행되어 진다.
  - 이를 통해 테스트 동안 결함이 발견되었을 경우, 개발 활동의 어느 단계가 재 수행되어야 하는지 파악 할 수 있다.

2. VP 모델 (V model with Prototyping)
  - 프로토타이핑은 시스템에 대한 이해 또는 리스크, 불확실성 요소와 같은 이슈를 해결하기 위해 시스템 혹은 시스템의 일부분을 빠르게 개발하는 방법
  - 무엇이 필요하고 무엇이 개발되어야하는지 개발자와 고객이 공통적인 이해를 이끌어낼 수 있다.
  - 프로토타이핑은 폭포수모델 혹은 V모델 등의 개발 단계에 적용하거나 독립적인 생명주기 모델로도 사용가능하다.
  - V 모델에 프로토타이핑 기법을 추가함으로써 프로젝트의 불확실성 요소나 리스크를 줄일 수 있다.
  - 불확실성 요소나 리스크로는 요구사항 단계에서의 요구사항 구현 가능성 여부, 설계 단계에서의 시스템 성능 검증여부, 새로운 도구의 도입 및 외주 개발 시스템의 위험 요소 등이 있다.

3. 점증적 모델 (Incremental Model)
  - 시스템 개발 시간을 줄일 필요가 있는 경우 유용하다.
  - 핵심이 되는 부분을 먼저 개발 한 후 나머지 기능을 구현하는 방식
  - 대부분의 요구사항이 정의되어 있지만, 시간이 지남에 따라 개선 여지가 있는 경우에 유용한 모델이다.
  - 개발 초기의 외부 인터페이스에 대한 리스크를 줄이는데 유용하게 적용될 수 있다.
  - V모델 및 VP 모델 모두 적용가능하다.

4. 진화 모델 (Evolutionary Model)
  - 시스템 개발 시간을 줄일 필요가 있는 경우 유용하다.
  - 전체 시스템에 대한 개발 단계가 여러번 반복되며 각 시스템은 사용자에게 모든 기능을 제공한다.
  - 하나의 시스템이 개발되어 사용되면서 변경 사항이 도출되고 이 변경사항은 다음 시스템 개발에 반영된다.
  - 시스템 명세가 전체적으로 불분명한 경우, 제품의 개선이 지속적으로 요구되는 경우에 적합한 모델이다.
  - 실무에서는 보통 점증적 모델과 조합해서 사용하는 경우가 많다. 새로운 시스템 버전에서는 기존 기능이 향상됨과 동시에 새로운 기능이 추가 될 수 있다.
  - 시스템이 완전한 기능을 갖추고 있지 않을지라도 시스템에 대한 초기 교육이 가능하다.
  - 개발 기간의 단축으로 인한 시스템의 조기 사용으로 경쟁력이 강화 될 수 있다.
  - 시스템이 여러 번에 걸쳐 개발되기때문에 시스템의 예상치 못 한 문제 도출 및 조기 수정이 가능하다.

# 소프트웨어 개발 방법론

- 구조적 방법론
  * 업무활동 중심의 방법론
  * 추상화, 구조화, 단계적 상세화, 모듈화
  * 특징
    + 분활과 정복의 원칙
    + 프로젝트 로직 중심
    + 컨트롤 가능한 모듈로 구조화
  - 주요 산출물
    - 도메인 분석서
    - 데이터 흐름도
    - 구조도
    - 프로그램 사양서
  - 지원도구 : Teamwork, SA
  - 주요 지원 언어 : COBOL, C, VB, PASCAL
2. 정보공학 방법론
  - 데이터 중심의 방법론
  - 정보전략계획, 업무영역분석, 업무시스템설계, 시스템구축
  - 특징
    - 기업업무 지원시스템, 지원 방법론
    - 데이터모델 중시
    - 프로그램 로직은 데이터 구조에 종속(CRUD)
    - 전사적 통합 데이터 모델
  - 주요 산출물
    - 도메인 분석서
    - ERD
    - 기능차트
    - 어플리케이션 구조도
    - 프로그램 사양서
    - 테이블 정의서 / 목록
  - 지원도구 : Cool Gen, SA
  - 주요 지원 언어 : COBOL, C, VB, PASCAL

3. 객체 지향 방법론
  - 객체, 클래스 간의 관계를 식별하여 설계모델로 변환하는 방법론
  - 요건정의, 객체 지향 분석(객체 모델링, 동적모델링, 기능모델링)
  - 객체 지향 설계, 테스트/배포
  - 특징
    - 프로그램 단위는 객체
    - 데이터와 로직 통합
    - 고도의 모듈화
    - 상속에 의한 재사용 분석
    - 설계간 갭(gap) 없음
  - 주요 산출물
    - 비즈프로세스 / 개념도
    - 다이어그램 (유스케이스, 시퀀스, 클라스, 컴포넌트 등)
  - 지원도구 : Rose, SA, Plastic
  - 주요 지원 언어 : C++, JAVA, VB

4. CBD 방법론
  - 재사용이 가능한 컴포넌트의 개발 / 상용 컴포넌트들을 조합하는 방법론
  - 요구분석, 분석(아키텍쳐 정의, 유스케이스 모델링), 설계, 개발, 구현(릴리즈, 교육)
  - 특징
    - 객체 방법론의 진화
    - 인터페이스 중시
    - 인터페이스의 구현은 컴포넌트
    - 블랙박스 재사용 지향
  - 주요 산출물
    - 비즈프로세스 / 개념도
    - 다이어그램 (유스케이스, 시퀀스, 클라스, 컴포넌트 등)
    - 재사용 계획서
    - .Net, EJB
  - 지원도구 : Cool Joe, Together
  - 주요 지원 언어 : 원칙적으로 개발언어 무관!
