# 정보보안의 목표 (3원칙)
- 기밀성, 무결성, 가용성
- 기밀성 : 정보를 누설하지 않는 것 비밀성 보장, 중요한 내용이 다른 사람에게 노출 되지 않도록 하는것.(암호화)
- 무결성 : 정보를 전송하는 과정에서 불법적으로 생성, 변경, 삭제되지 않고 원본이 유지되는 것. 위변조가 없었음을 보증.(해쉬함수[^1])
- 가용성 : 인가받은 사용자에 대하여 요구한 정보를 필요한 시점에 접근 및 사용을 보장해 주는 것을 의미한다.(DDos 공격으로 침해당할 수 있다.)

[^1]: 또는 해쉬 알고리즘. 가변길이 문자열을 고정길이 문자열로 바꿔줌. 메세지 다이제스트를 생성(해쉬값). 단방향으로 전송된다. 역방향으로 원본데이터를 얻기위해서는 기회비용이 딸림. 정보보안, 인공지능, 블록체인, 데이터베이스 등에 사용된다.

# 암호기술의 분류

### 대칭키
- 비밀키,
- 암호화 키 = 복호화 키,
- 키의 수 = n(n-1)/n (pm복)
- 연산속도가 빠르다,
- 키를 공유하기 어렵다.
- 블록 암호 : 평문에 블록 단위로 암호화를 수행
- 스트링 암호 : 평문을 비트 단위로 암호화를 수행, 컴퓨팅 파워가 부족한 임베디드 환경에서 사용한다. 요즘은 거의 안 쓴다.
- 암호 알고리즘은 RNN이랑 비슷하다.

### 비대칭키
- 공개키,
- 암호화 키 != 복호화 키
- 키의 수 = 2n
- 대칭키의 키를 안전하게 공유하기 위해 개발되었다.
- 공인인증서 공개키는 엘뎁스 서버에 저장
- 개인키로 암호화한것을 공개키로 복호화(전자서명, 부인 방지)
- 공개키로 암호화한 것을 개인키로 복호화(이체 등)
- 단문에 문자열을 암호화 할 때 사용

### 해쉬
- 해쉬함수, 해쉬알고리즘
- 가변길이 문자열을 고정길이 문자열로 변환
- 역방향으로 계산적으로 너무 오래걸린다.
- 단방향 암호 알고리즘
- 비밀번호
- 개인정보 보호법 준수를 위한 부록 - 비밀번호는 단방향 알고리즘을 사용해야한다.
- 바이오 정보는 단방향 알고리즘을 꼭 사용하지 않아도 된다.
- SHA 256 : 모든 문자열을 256자로 바꿔준다. 그러다보면 서로 다른 값끼리 충돌하는 경우가 생김. 이런 취약점을 이용해서 공격하는 방식은 Birthday 공격
- 원문과 해쉬값으로 이루어진 테이블을 엄청 길게 만들어서 전부 대입하면서 공격하는 방식 Rainbow 공격
- 쓰면 안될 해쉬함수 : MD4, MD5, SHA1
- SHA256 이상의 해쉬함수를 사용해야한다.(~~똑똑한 신입사원~~)
- SALT : 비밀번호 + __임의의 문자열__, Rainbow 공격을 막을 때 사용할 수 있다.

### APT 공격(Advenced Persistent Threat)
- 오랫동안 정보를 수집
- 지능화된 공격 기법 사용

### 파밍(Pharming)
- DNS 공격
  - Domain : 사람이 인식 할 수 있는 논리적인 주소
  - IP : 컴퓨터가 인식 할 수 있는 논리적인 주소, 위변조가 가능함,
  - MAC : 전세계 제조사가 유니크한 값을 가진다
  - DNS 서비스 : Domain - IP 변경해주는 서비스
  - ARP 프로토콜 : IP주소를 맥주소로 바꿔줌
  - RARP 프로토콜 : 맥주소를 IP주소를 바꿔줌
  - 스푸핑 : 무결성 침해, 바꾸기
  - 스니핑 : 기밀성 침해, 엿보기
  - 컴퓨터가 로컬에 있는 Domain과 IP를 먼저 검색한다는 점을 이용한 공격
    - 리눅스 : /etc/hosts
    - 윈도우 : C:\windows\system32\drivers\hosts

### 큐싱
- QR코드로 하는거

### 스미싱
- 단문 문자를 보내서 특정 주소로 유도하는 방식

### Drive by download 공격
- 접속하자마자 다운로드를 받는 공격

### 스피어 피싱 (Sphear fishing)
- 딱 찍어서 그 사람한테 공격하는 방식
- 그 사람에 대한 정보를 너무 잘 알고 있기때문에 걸려들기 쉬운 방식으로 접근한다.

### 랜섬웨어

- 항상 백업을 잘하는 습관을 갖자

# 시큐어 코딩
- 보안은 무조건 고려해야한다.
- 취약점이 발견되어 런칭을 하지 못하는 경우가 있을 수 있음

### 시큐어 개발 생명주기 (Secure SDLC)
- 예전에는 코딩단계에서만 시큐어 코딩을 진행함
- 개별 코드는 안전하더라도 전체적인 아키텍쳐의 보안 취약점이 생길 수 있음
- 기본적으로 보안에 필요한 요구사항 등을 다루어야한다.
  - 요구사항 정의서, RFP, 제안서는 가장 중요한 문서이므로 잘 관리하자
- 법적 규제 등등
- 정적 테스트 : 화이트 박스 테스트 - 소스코드 레벨의 보안 테스트
  - 포티파이 등 툴을 이용하여 테스트를 진행한다.
- 동적 테스트 : 블랙 박스 테스트 - 소스를 가리고 점검
  - 침투 테스트라고도 한다.

### SQL 삽입 공격 대응을 위한 시큐어 코딩 기법
- UI - 1차 방화벽 - 웹 서버(DMZ) - 2차 방화벽 - AP 서버 - 3차 방화벽 - DB
- ~~중국에서 SQL 인젝션 도구 만원에 판다고 한다.~~
- 파싱 : 컴퓨터가 문자열이 들어왔을때 로직트리를 만드는 과정
- 하드 파싱 : 1억개의 쿼리가 전부 다른 쿼리라고 인식하는 것, DBMS 성능 저하와 인젝션 공격에 취약하다.
